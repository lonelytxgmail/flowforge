# FlowForge Development Roadmap

这份文档用于收敛 FlowForge 后续开发计划。

目标不是把所有想法一次性做完，而是按照“单人可维护、公司可用、后期可扩 AI/Agent”的原则，分阶段推进。

## 1. 总体原则

后续开发持续遵守以下原则：

1. `Workflow DSL` 是唯一真相源。
2. 表单编辑、JSON 编辑、图形化画布，都是 DSL 的不同编辑视图。
3. 运行时状态以数据库持久化模型为准，不依赖日志拼装。
4. 新节点扩展优先通过“节点类型 + 执行器 + 配置 schema”完成，而不是在引擎里硬编码分支。
5. 第一阶段不把 Redis、ES、MQ 作为平台强依赖；它们是节点能力，不是平台内核。
6. 设计始终优先单人长期维护，避免引入高复杂度抽象。

## 2. 当前状态

截至目前，项目已经具备这些基础能力：

- 后端模块化单体骨架已跑通
- 工作流定义、版本发布、实例启动可用
- 运行态持久化模型已落地
- `workflow_task` + worker 驱动执行模型已可运行
- 已支持节点：
  - `START`
  - `END`
  - `ATOMIC_ABILITY`
  - `DIGITAL_EMPLOYEE`
  - `CONDITION`
  - `WAIT_FOR_FEEDBACK`
- 已支持基础运行控制：
  - 暂停
  - 恢复
  - 重试
  - 提交反馈
- 前端管理台第一版已联调通过
- 中文默认管理台已可用

## 3. 后续阶段划分

## 阶段一：收口第一阶段主链路

目标：

- 把“定义 -> 发布 -> 启动 -> 运行 -> 反馈 -> 查看”的主链路打磨完整
- 让管理台从“演示可用”进入“持续可用”

当前阶段的优先事项：

1. 管理台完善
   - 工作流列表与实例列表
   - 搜索、筛选、状态过滤
   - 空态、错误态、加载态统一
   - 工作流详情与实例详情继续打磨

2. DSL 可用性增强
   - 输出 DSL 规范文档
   - 输出节点目录文档
   - 新建页和详情页继续增强模板、校验、格式化
   - 明确变量替换规则

3. 等待反馈闭环完善
   - 前端更清晰展示“等待人工反馈”
   - 反馈记录展示结构化内容
   - 等待态与恢复态的提示更清晰

阶段一完成标准：

- 新用户可以看文档和管理台提示，理解 DSL 的基本格式
- 能独立创建一个简单工作流并启动实例
- 能清楚看懂实例状态、节点执行、事件和反馈链路
- 常见错误不会只返回模糊报错

## 阶段二：运行内核增强

目标：

- 把执行引擎从“基础可跑”增强到“稳定可控”

重点内容：

1. 任务模型增强
   - `workflow_task` 查询 API
   - 按实例查看任务链路
   - 更细粒度的失败任务重试

2. 状态机约束增强
   - 实例状态迁移收紧
   - 节点状态迁移收紧
   - Worker 抢占和重复消费保护

3. 重试与超时
   - 节点级重试策略
   - 超时配置
   - 错误分类

4. 审计增强
   - 统一事件类型
   - 统一事件 detail schema
   - 更清晰的错误输出

阶段二完成标准：

- 失败和重试行为可解释、可追踪、可恢复
- 任务链路能清楚反映每次执行和重试
- 运行异常不会导致状态混乱

## 阶段三：节点能力产品化

目标：

- 把当前“代码里支持”的节点能力，升级成“管理台可配置、可理解、可复用”的产品能力

建议顺序：

1. `REST`
2. `DATABASE`
3. `DIGITAL_EMPLOYEE`
4. `REDIS`
5. `ELASTICSEARCH`
6. `KAFKA`

每类节点都要具备：

- 配置 schema
- 文档说明
- 前端表单
- 示例模板
- 运行结果展示
- 基础校验

这里最重要的是先把 `REST` 节点做强，因为它会承载很多接入场景：

- 普通 HTTP 调用
- 登录后复用 session
- token / cookie 提取
- 上下文透传
- 流式与非流式

阶段三完成标准：

- 主要原子能力不再依赖手写 JSON 猜格式
- 使用者可通过模板和表单稳定配置节点

## 阶段四：DSL 规范化与结构化编辑

目标：

- 把 DSL 从“可写”推进到“易写、可验证、可演进”

重点内容：

1. DSL 规范文档
   - 顶层结构
   - 节点结构
   - 边结构
   - 变量规则
   - 条件表达式规则

2. 节点目录文档
   - 每种节点类型的配置字段
   - 示例
   - 常见错误

3. 结构化编辑器
   - 左侧表单
   - 右侧 JSON 预览
   - 模板插入
   - 发布前校验

4. 变量替换体系
   - `input`
   - `context`
   - `steps`
   - `system`

建议变量规则：

```text
{{ input.orderId }}
{{ context.auth.token }}
{{ steps.review_1.output.approved }}
{{ system.instanceId }}
```

阶段四完成标准：

- 普通用户可以不直接手写整份 DSL
- DSL 发布前可以发现绝大部分结构性错误

## 阶段五：图形化编排器

目标：

- 在不改变 DSL 真相源的前提下，引入图形化拖拽编排

关键原则：

1. 图形化画布不是新的定义模型
2. 画布只是一种 DSL 编辑器
3. 画布布局信息和执行语义分离

推荐实现方式：

- 前端使用 `React Flow` 一类画布库
- 中间画布只负责节点与连线
- 右侧属性面板负责编辑 `config`
- 最终统一生成并保存 DSL

图形化实现拆分为两个子阶段：

### 5.1 可视化查看

- 用画布展示节点和边
- 支持缩放、定位、基础高亮
- 支持从 DSL 渲染成图

### 5.2 基础拖拽编辑

- 拖拽创建节点
- 连接边
- 编辑节点属性
- 保存回 DSL

布局信息建议单独存储：

- `layout_json`
- 或 DSL 内部单独的 `ui` 区域

这里不要把以下内容混进执行模型：

- 节点坐标
- 折叠状态
- 分组样式
- 视图缩放

阶段五完成标准：

- 用户可通过画布查看与编辑流程
- 画布与 DSL 可双向稳定转换
- 图形化不会破坏后端执行模型

## 阶段六：可运维性与团队化准备

目标：

- 在仍保持单人可维护的前提下，增强内部使用的可靠性

重点内容：

1. 文档体系整理
   - 架构文档
   - DSL 文档
   - 节点文档
   - 本地开发文档

2. 测试体系整理
   - 后端单元测试
   - 执行引擎集成测试
   - 前端关键页面冒烟
   - 本地 e2e 脚本

3. 环境配置规范
   - `docker compose` 完整本地环境
   - `.env` 规范
   - profile 规范

4. 审计与操作安全
   - 更清晰的操作日志
   - 关键操作确认机制
   - 基础权限模型预留

## 阶段七：第二阶段 AI / Agent 预留

目标：

- 只做正确的预留，不提前做复杂多 Agent 协作

优先内容：

1. `LLM` 节点配置骨架
2. `Tool` 节点统一抽象
3. `Agent` 节点统一抽象
4. 工具调用结果标准输出模型
5. 模型输入输出 schema 占位

当前不做：

- 复杂多 Agent 协作
- 长期记忆
- 复杂评测体系
- 一开始就微服务化

## 4. 新节点扩展规则

后续要确保新增节点足够容易扩展，这一点必须作为长期约束。

新增节点时，统一遵守以下步骤：

1. 定义节点类型
   - 新增 `NodeType`
   - 定义该节点的语义边界

2. 定义配置 schema
   - 明确 `config` 里允许哪些字段
   - 区分必填项和可选项

3. 定义执行器
   - 新增 `NodeExecutor` 实现
   - 不把节点业务逻辑塞进通用引擎主流程

4. 定义文档与模板
   - 增加 DSL 示例
   - 增加节点说明文档
   - 增加前端模板

5. 定义前端编辑方式
   - 表单字段
   - JSON 预览
   - 校验提示

也就是说，扩展节点的标准产物应该是：

- 后端 `NodeType`
- 后端 executor
- 配置 schema
- 示例 DSL
- 前端模板
- 前端表单
- 文档

如果未来新增节点仍然只靠“手写 JSON + 后端埋逻辑”，维护成本会很快失控。

## 5. 推荐的短期执行顺序

建议后续按下面顺序推进：

1. 完成阶段一剩余收口
   - DSL 文档
   - 节点目录
   - 变量规则

2. 进入阶段二
   - 运行内核增强
   - 任务查询和更细粒度重试

3. 进入阶段三
   - `REST` 节点产品化
   - `DATABASE` 节点产品化
   - `DIGITAL_EMPLOYEE` 节点产品化

4. 再进入阶段四和阶段五
   - 结构化编辑
   - 图形化编排器

## 6. 当前结论

当前项目应坚持的方向是：

- 先把 DSL 和节点 schema 做稳
- 再把结构化编辑做好
- 最后再上图形化拖拽

这样后续新增节点、增加图形化能力、引入 AI/Agent，都不会推翻当前架构。
