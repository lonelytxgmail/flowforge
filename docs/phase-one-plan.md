# FlowForge Phase One Plan

这份文档只针对“第一阶段”收口计划。

第一阶段的目标不是把平台做成最终形态，而是交付一个：

- 可运行
- 可理解
- 可继续扩展
- 对单人维护友好

的基础版本。

## 1. 第一阶段目标

第一阶段明确聚焦以下能力：

1. 支持工作流/策略编排
2. 支持原子能力节点编排
3. 支持数字员工调用节点
4. 支持执行日志、执行明细、反馈链路
5. 支持基础运行态管理
   - 启动
   - 执行
   - 失败
   - 重试
   - 暂停
   - 恢复
6. 保持模块化单体架构

第一阶段不做：

- 复杂多 Agent 协作
- 长期记忆
- 复杂评测体系
- 微服务化拆分
- 把 Redis / ES / MQ 做成平台核心依赖

## 3. 第一阶段剩余工作

第一阶段接下来建议分 4 个子阶段完成。

## 3.1 文档与规范收口

目标：

- 让 DSL 和节点配置不再依赖“猜”

产出：

- `dsl-spec.md`
- `node-catalog.md`
- 示例 DSL 清单

完成标准：

- 新用户能知道 DSL 顶层结构
- 能知道每种节点如何配置
- 能知道参数和变量从哪里来

## 3.2 管理台可用性增强

目标：

- 从“能演示”提升到“能日常使用”

产出：

- 工作流列表和实例列表筛选完善
- 详情页提示完善
- JSON 编辑错误提示完善
- 发布和启动流程更顺畅

完成标准：

- 普通用户不需要频繁看后端代码就能完成基本操作

## 3.3 运行内核增强

目标：

- 把运行态控制补得更扎实

产出：

- 任务级查询能力
- 更细的失败任务重试
- 节点超时与重试策略预留
- 更清晰的事件与错误输出

完成标准：

- 失败、暂停、恢复、等待反馈行为都可解释、可追踪

## 3.4 节点能力产品化起步

目标：

- 把核心节点配置从“手写 JSON”推进到“模板 + 表单 + JSON 预览”

建议顺序：

1. `REST`
2. `DATABASE`
3. `DIGITAL_EMPLOYEE`
4. `CONDITION`
5. `WAIT_FOR_FEEDBACK`

完成标准：

- 至少最常用节点可以通过结构化方式配置

## 4. 第一阶段开发顺序

建议严格按这个顺序推进：

1. 完成 DSL 文档与节点目录
2. 明确变量替换规则
3. 再推进前端结构化编辑
4. 再做运行内核增强
5. 最后做阶段一整体测试与文档整理

原因：

- DSL 和节点 schema 不清晰时，前端结构化编辑很容易返工
- 运行内核和节点产品化都依赖一套稳定的定义规范

## 5. 第一阶段验收标准

第一阶段最终应满足：

1. 可以在管理台中创建工作流、发布版本、启动实例
2. 可以查看实例状态、节点执行、事件日志、反馈记录
3. 可以完成等待反馈型流程
4. 用户能通过文档理解 DSL 基本结构和节点配置格式
5. 新增节点时有明确扩展路径，不需要修改大量核心代码

## 6. 第一阶段之后的衔接

第一阶段完成后，再进入：

- 运行内核更强的增强
- 节点产品化
- 图形化编排器
- AI / Agent 预留扩展

也就是说，第一阶段不是终点，而是后续图形化和 AI 扩展的稳定底座。
